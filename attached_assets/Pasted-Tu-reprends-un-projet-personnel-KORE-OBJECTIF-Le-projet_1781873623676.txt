Tu reprends un projet personnel KORE.

OBJECTIF
- Le projet reste personnel pour l’instant.
- On garde une base propre pour pouvoir le transformer en produit plus tard si la valeur se confirme.
- Toute amélioration doit rester utile, simple, robuste et livrable.
- Tout doit être poussé sur la branche main.
- Si tu ne peux pas pousser sur main, le lot ne compte pas.

RÈGLE ABSOLUE
- Avant toute modification, vérifie la branche active.
- Vérifie l’état exact du repo.
- Compare avec main.
- Si nécessaire, rebase sur main.
- Puis seulement : cartographie, plus petit fix utile, test, push.
- Après un push réussi, passe au lot suivant.
- Si le push échoue, stoppe-toi et documente le blocage au lieu de continuer.

MODE RED TEAM
- Commence toujours par cartographier le problème.
- Cherche le bug réel ou le faux signal réel.
- Choisis le plus petit fix utile possible.
- Ne fais jamais de refactor global caché dans un correctif ciblé.
- Ne transforme jamais un lot en mini-reconstruction du produit.
- Ne touche jamais au périmètre hors lot sans validation explicite.
- Ne rajoute aucun “bonus” opportuniste.

PRIORITÉS DE TRAVAIL
1. Fiabilité des données.
2. Visibilité des signaux utiles.
3. Propreté TypeScript.
4. Portabilité / build / install.
5. Corrections de bugs visibles.
6. Extensibilité seulement si elle ne dégrade pas la simplicité.

LOT 1 — CONFIANCE DES DONNÉES DU DASHBOARD
Cartographie :
- Vérifier la source du timestamp de fraîcheur.
- Vérifier si le dashboard affiche la vraie génération du briefing ou seulement l’heure du fetch client.
- Vérifier si des signaux affichés peuvent être trompeurs.

Fix minimal attendu si le problème existe encore :
- Remplacer le faux signal de fraîcheur par le vrai timestamp de génération du briefing.
- Utiliser le champ serveur correct s’il existe déjà.
- Ne rien changer d’autre au dashboard.

Validation :
- typecheck
- build si nécessaire
- push sur main
- puis seulement continuer

Validation :

LOT 3 — PROPRETÉ TYPESCRIPT
Cartographie :
- Relever les `as any`, `as BriefingExtended`, casts inutiles, champs fantômes, et schémas incomplets.
- Vérifier si un cast masque un vrai désalignement de type.
- Vérifier si un type peut être complété proprement au lieu d’être forcé.

Fix minimal attendu si le problème existe encore :
- Supprimer uniquement les casts qui cachent une erreur réelle.
- Compléter le schéma si les champs existent réellement côté API.
- Ne pas toucher aux fichiers hors périmètre si ce n’est pas indispensable.

Validation :
- typecheck obligatoire
- build si pertinent
- push sur main
- puis seulement continuer
